# BÁO CÁO KIỂM TOÁN MÔ-ĐUN USER (COVERAGE AUDIT)

**Mục tiêu:** Ánh xạ các thực thể với API, rà soát vùng trũng dữ liệu, kiểm tra bảo mật (RBAC) và phát hiện chai cổ chai hiệu năng (như N+1 Query).

Do mô-đun User là trung tâm phân quyền và định danh cho toàn bộ thao tác của Front-end và Back-office, tài liệu này đặc tả chi tiết về cách hệ thống tương tác và đề xuất khắc phục những lỗ hổng kỹ thuật vô hình.

Cách Service trả về tập quyền Context (Quyền quản trị vs Quyền cá nhân) rất thông minh.                                                                                                                                                                               | Tận dụng API `/users/me/access-context` ở Frontend (khi App Init) để store vào Vuex/Redux, dùng để ẩn/hiện Button theo Authority thay vì parse Token chay.                                                                                                                    |

### SUY LUẬN & KẾ HOẠCH TRIỂN KHAI CHO FRONT-END:

_Ghi chú dành riêng cho Leader / Front-end:_

1. **Lỗi `date_of_birth`:** Trong lúc tôi viết Ticket để sửa file `UserMapper.java` ở tầng Backend, anh em FE Frontend khoan hãy build chức năng "Ngày Sinh" trong `/profile` để tránh user khiếu nại (vì nhấn lưu sẽ bị vô tác dụng).
2. **Quyền hạn Admin:** Tóm gọn `access-context` cung cấp cả `roles`, `permissions` lẫn mảng đếm `managementRoles`. Ở FE (React/Vue/Next), làm một `RequireAuth` Guard lấy state `hasManagementAccess=True` mới cho vào route `/admin`. Đừng soi JWT payload làm gì cho cực, gọi endpoint đó cho chắc chắn.
3. **Hibernate Memory Bug:** Chức năng liệt kê user (Admin List) nếu gặp chậm và CPU nhảy 100% thì báo tôi ngay. Nguyên nhân nằm ở `FetchType.EAGER` của User Role. Đó là bom nổ chậm do Dev trước để lại, cần đưa vào Next Sprint để refactor thành `FetchType.LAZY` luôn.
